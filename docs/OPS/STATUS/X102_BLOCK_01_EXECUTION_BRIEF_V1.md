## 1. Назначение
Этот документ является прямым исполнительным ТЗ для агента `Codex`.

Агент `Codex` должен:
1. сначала проверить полноту handoff по `X102`;
2. только после успешной проверки начать кодовую работу;
3. выполнить только `Блок 01`;
4. изменить только разрешенные файлы;
5. вернуть результат строго по packet-формату.

---

До старта кодовой работы `Codex` обязан найти и провалидировать:

## 4. Что входит в Block 01
`Codex` имеет право работать только с верхней управляющей композицией.

В `Block 01` входят:
1. верхняя рабочая полоса;
2. плавающая панель инструментов;
3. левая группа значков верхней зоны;
4. кнопка вызова компактной панели настроек верхней зоны;
5. компактная панель настроек верхней зоны;
6. поведение привязки панели;
7. общая геометрия верхней композиции.

---

## 5. Что не входит в Block 01
`Codex` не имеет права менять:

1. центральную рабочую область;
2. нижнюю строку состояния;
3. правую боковую зону;
4. модальные окна;
5. вторичные всплывающие панели вне верхней зоны;
6. ядро;
7. миграции;
8. восстановление;
9. формат хранения;
10. stage-gated логику;
11. сетевую модель;
12. delivered baseline функционал вне реально затронутой поверхности.

Если для literal match требуется выход за эти границы:
- остановиться;
- вернуть `HOLD`;
- описать, какой scope extension нужен от `Docs`.

1. обязательны для текущей визуальной приемки;

3. не запрещают будущий редизайн в следующих контурах;
4. не переопределяют активный канон по границам и запретам.

### 6.3 Правило приоритета

1. `master screen`

Если спор идет о конкретной зоне:

## 12. Критерии приемки Block 01

### 12.1 Визуальные
1. верхняя рабочая полоса совпадает с эталоном;
2. плавающая панель инструментов совпадает с эталоном;
3. левая группа значков совпадает с эталоном;
4. кнопка вызова и компактная панель настроек совпадают с эталоном;
5. геометрия верхней композиции совпадает с эталоном.

### 12.2 Поведенческие
1. плавающая панель перемещается правильно;
2. привязка к верхней зоне работает правильно;
3. элементы управления остаются рабочими;
4. компактная панель настроек открывается и закрывается правильно;
5. верхняя зона не мешает работе редактора.

### 12.3 Технические
1. сборка проходит;
2. обязательные проверки проходят;
3. новые зависимости не появились;
4. запрещенные файлы не изменены;
5. ядро не затронуто;
6. stage-gated правила не нарушены;
7. ограничения локального режима и сети не нарушены.

Отдельное правило для `merge-readiness`:
- проверка обязательна;
- она не должна трактоваться как требование `current branch = main` для обычного execution на рабочей ветке;
- она должна исполняться только в `main-compatible preview context` или в `PR-closeout context`.

### 12.4 Процессные
1. ручная визуальная проверка пройдена;
2. execution-status синхронизирован с кодом;
3. packet-state не отстал от repo-state;
4. переход к следующему блоку не разрешается автоматически.
